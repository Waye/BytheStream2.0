## 2026/4/21 晚 — 批次 9：性能诊断全过程 + 修 3 处跳转 bug + 数据库索引 + README 加截图

### 总览

批次 8 跑通 OAuth 之后发现点 MiniPlayer 文章标题跳文章页非常慢（经常永远转圈）。以为是前端性能或网络问题，花一小时排查后发现真因是 **数据库没索引 + 三处跳转 URL 拼错**。本条记录完整诊断过程和修复路径。

### 诊断的假设 → 验证 → 排除

一开始猜了很多方向，挨个排除：

1. **Apollo cache 没命中** → 验证：`lib/graphql/client.ts` 全局默认已是 `cache-first`。排除。

2. **MiniPlayer 没 prefetch** → 思路：音频加载时顺手 `client.query(GET_ARTICLE, ...)` 把文章预取进 Apollo cache。写了补丁但因为 MiniPlayer 里的锚点字符串和实际代码不匹配，Python `str.replace` 静默失败。**有用但没打进去**。

3. **React 渲染慢** → Chrome DevTools Network 显示所有 GraphQL 请求都是毫秒级。排除。

4. **Metro bundler 断连** → Chrome Console 确实看到 `Disconnected from Metro (1006: "")`。`pkill -f expo && pkill -f metro` 全部重启。恢复正常但文章页**还是慢**。

5. **Webpack 开发模式慢** → 看 Network 时间轴：实际请求集中在前 500ms，但中间**有 2.41 秒纯空白没发请求**。说明网络不是瓶颈，有东西卡在 JS 线程。

6. **MongoDB 查询慢** → ✅ **真因**。用 `mongo` CLI 看 `Articles` 索引：

```javascript
// 现有索引:
{"_id":1} _id_
{"_fts":"text","_ftsx":1} category_text_title_text_author_text_content_text
{"id":1,"volume":1} id_1_volume_1_autocreated
```

没有 `{volume:1, slug:1}` 复合索引。而 resolver 查 `article(volume, slug)` 就用的是这两个字段。每次全表扫 1831 条文档，所以第一次查任何新文章都是 2.4 秒。

**修复**：`server/check_idx.ts` 跑一次 `createIndex({volume:1, slug:1})`，耗时 50ms，查询时间从 2.4s → 10ms。

### Bug 2：三处跳转 URL 拼错

索引修完了，但前端有些 URL 仍然查不到文章（GraphQL 返回 `null`，页面一直转圈）。继续挖：

**bug 2a — MiniPlayer 点标题跳文章** (`lib/ui/MiniPlayer.tsx:241`):
```tsx
// 错误
const slug = String(track.id).replace(/^\d+:/, '');  // 砍掉 volume 前缀!
router.push(`/article/${slug}`);

// 正确
router.push(`/article/${encodeURIComponent(String(track.id))}`);
```

把 `85:11_li_s` 砍成 `11_li_s`。article 页解析 `id` 时拿不到 volume，只能兜底取 1，自然查不到 85 期的文章。

**bug 2b — 首页队列 stack 跳转** (`app/index.tsx:220`):

看 grep 结果时以为这行和别处不一样（`.replace(/^\d+:/, '')`），但实际后来发现已经被改对了。虚惊一场。

**bug 2c — 文章页 article=null 时永远转圈** (`app/article/[id].tsx:133`):
```tsx
// 错误
{loading || !article ? <Spinner /> : <Content />}
// loading=false 但 article=null 时继续显示 spinner,看起来像卡死

// 正确
{loading ? <Spinner />
 : !article ? <NotFound 找不到这篇文章 + 返回按钮 />
 : <Content />}
```

这个 bug 的危害是：任何脏 URL (比如 `/article/1_prayer_s` 没 volume 前缀) 都会让文章页看起来永远在加载，不给用户任何反馈。

### Bug 追溯：脏 URL `1_prayer_s` 是怎么跑出来的

查了 `curl -X POST /graphql {article(volume:85, slug:"1_prayer_s")}` → 返回 `null`。再查 `Articles.find({slug:"1_prayer_s"})` → **0 条**。这个 slug 数据库根本没有。

推测是某个列表渲染（收藏？推荐？最近播放？）的 id 拼错了。既然 MiniPlayer 的 track.id 在 bug 2a 之前会被切成裸 slug 推给 router，点击后 article 页只得到 `slug=1_prayer_s, volume=1(兜底)`，数据库里第 1 期也没这篇，返回 null。修完 bug 2a + 2c 后这类脏 URL 就算进到 article 页也会显示"找不到"而不是卡住。

`assets/pics/` 下已有 ios1-9.png + web1-3.png（用户拍的）+ church-old/scarborough/toronto.jpg（教堂照片）。合计 ~9 MB。

- 用 `pngquant --quality=65-80 --ext=.png --force *.png` 压缩到 ~2-3 MB

### 今天的最终状态

- **文章页加载性能**：从 2.4s → ~200ms（索引 + 跳转 URL 修正）
- **健壮性**：所有 `/article/xxx` 脏 URL 都有视觉反馈而不是卡死
- **OAuth**：Google/Facebook 全跑通（批次 8 已完成）

### 待做

- 真正跑起来 prefetch (MiniPlayer 加载音频时顺手 prefetch GET_ARTICLE)，让点标题的感受从"200ms 可见延迟"变成"瞬间"
- 把性能监控接入后端：慢查询日志 + Sentry（可选）
- 清理数据源的脏 id（追到推荐算法/历史记录哪里给了 `1_prayer_s`）

### 教训

1. **先看数据库索引**。Web 应用慢 80% 的情况不是前端渲染慢，是后端 SQL/MongoDB 查询慢，而查询慢 80% 是缺索引。以后遇到首次加载慢先 `db.collection.getIndexes()`。

2. **Python heredoc 不适合改 JS 字符串**。一堆反斜杠会被 shell 或 Python 双重转义吃掉。之后改代码改用 `node -e` 或直接写 `.js` 文件跑。

3. **`loading || !article ?` 是反模式**。应该拆成三态：loading / error / not-found / content，每种给用户不同反馈。默默转圈是最糟糕的体验。

## 2026/4/20 — 批次 7：音频系统全线打通 + TTS 本地批处理

### MeloTTS 调试踩坑（按出现顺序）

1. **unidic 词典下载失败** — 网络在最后 1MB 卡死（S3 东京节点）
   - 解法：用已装的 `unidic-lite` 软链接到 `unidic/dicdir`

2. **NLTK cmudict 下载被拦** — SSRF 走 `198.18.0.18`（本机代理或 VPN 干扰）
   - 解法：手动 `curl` 从 GitHub raw 下三个 zip 到 `~/nltk_data/corpora/` 和 `~/nltk_data/taggers/`

3. **MeloTTS HParams 没有 `.get()` 方法**
   - 报错：`AttributeError: 'HParams' object has no attribute 'get'`
   - 解法：`synth.py` 里 `_pick_speaker` 用 `dict(spk2id)` 转换

4. **HuggingFace 模型下载极慢**（`/text-tools/melo-data/...`）
   - 解法：`export HF_ENDPOINT=https://hf-mirror.com`，每次新终端都要 export
   - 永久写入：`echo 'export HF_ENDPOINT=https://hf-mirror.com' >> ~/.zshrc`

5. **MPS device 不一致** — `expected scalar type Float but found Half`
   - 解法：`sed` 把 `melo/text/*_bert.py` 里 `device = "mps"` 全替换为 `pass # disabled`，强制 CPU

6. **state.json 一直为空** — `generate_one.py` 单篇脚本不写 state（只 batch.py 才写）
   - 解法：写了一个临时 Python 脚本，用 `ffprobe` 扫描 `output/volume_*/*.mp3` 补回 state.json

7. **后端 ESM 报错** `__dirname is not defined`
   - 原因：`server/` 是 `"type": "module"`，没有 CommonJS 的 `__dirname`
   - 解法：audio.ts 顶部 `const __dirname = path.dirname(fileURLToPath(import.meta.url))`

8. **Python 3.13 Homebrew externally-managed**
   - 报错：`× This environment is externally managed`
   - 解法：必须先 `conda activate python-mac-gpu`（不要用系统 Python 装包）

9. **brew cleanup truffle 权限报错** — 忽略，ffmpeg 已经装好可以用

10. **resolvers.ts 替换两次失败** — heredoc 单引号 + 中文括号 + 缩进差异
    - 解法：用更鲁棒的 `re.search` + 显式分隔符替换
